# Skill 创作器

> Write → Eval → Patch → Ship。写完即测，测完即修，修完即用。

---

## 核心理念

- **Skill ≠ Prompt**：Skill 是可被 Agent 自主发现、加载、执行的标准化能力单元
- **TDD 驱动**：先想清楚"什么算成功"，再写 Skill
- **Token 是稀缺资源**：三层渐进加载，不塞常识，一个例子就够
- **五种设计模式**：选对模式，事半功倍

---

## 输入

用户描述想要的 Skill 功能。如果描述模糊，用创作四步法引导澄清。

---

## Pipeline

```
需求澄清 → 选型 → 骨架 → 正文 → 自检 → 输出 → [可选] 触发 eval-skill
```

---

## Phase 1：需求澄清（创作四步法）

依次确认以下四项，任一不清楚则询问用户：

| # | 问题 | 目的 |
|---|------|------|
| ① | **触发时机**：什么场景下 Agent 应加载这个 Skill？关键词/症状/工具名？ | 写 description |
| ② | **输入输出**：需要用户提供什么？最终交付什么？ | 定义接口 |
| ③ | **核心流程**：3-7 步主要步骤？ | 写正文 |
| ④ | **边界规则**：什么不能做？异常怎么处理？ | 写约束 |

---

## Phase 2：选型（五种设计模式）

根据需求选择主模式（可组合）：

| 模式 | 核心价值 | 适用场景 | 关键特征 |
|------|----------|----------|----------|
| **Tool Wrapper** | 封装工具调用 | API/CLI 标准化 | 输入校验 → 调用 → 格式化输出 |
| **Generator** | 生成内容 | 模板/脚手架/代码生成 | 参数 → 模板 → 产出 |
| **Reviewer** | 检查质量 | 审查/校验/诊断 | 输入 → 逐项检查 → 报告 → 等待确认 |
| **Inversion** | 约束行为 | 安全/流程管控 | 禁止列表 + 反制表 + 不可协商声明 |
| **Pipeline** | 串联步骤 | 复杂工作流 | 步骤 A → B → C，每步可独立测试 |

**选型决策**：你的 Skill 核心价值是调用工具、生成内容、检查质量、约束行为、还是串联流程？

---

## Phase 3：骨架生成

### 目录结构

```
{skill-name}/
├── SKILL.md          ← YAML 头 + 正文
├── references/       ← 大段参考（按需加载，节省 token）
└── scripts/          ← 可执行脚本（如有）
```

### YAML 头部

```yaml
---
name: {小写+连字符，≤64字符}
description: {触发描述，≤1024字符，写清"做什么"+"什么时候用"}
allowed-tools: {预授权工具列表}
---
```

### Description 写法（CSO 原则）

**You MUST:**
- 只写触发条件，不总结内部流程
- 包含关键词/症状/工具名，帮助语义匹配
- 稍微"积极"一些，多列举触发场景

**Do NOT:**
- 不要暴露内部步骤（"先 A 再 B 再 C"）
- 不要太笼统（"帮助用户处理代码"）
- 不要太窄（只绑定一个特定框架名）

**对比示例：**

| ❌ 太笼统 | ✅ 精准触发 |
|-----------|------------|
| "帮助用户处理代码相关任务" | "Git commit 规范检查。当用户提到'提交代码'、'commit message'、'规范检查'时触发。" |

---

## Phase 4：正文编写

### 编写原则（Anthropic 最佳实践）

1. **祈使句** — 直接告诉 Agent 该做什么
2. **解释为什么** — 比堆砌 MUST/SHOULD 更有效
3. **≤ 500 行** — 超出拆到 references/
4. **不写常识** — Agent 已知的不需要教
5. **一个例子够** — Claude 能自己迁移
6. **保持通用** — 理论指导代替硬编码特例

### 按模式编写

#### 约束型（Inversion）必备元素

```markdown
## 核心规则（不可协商）

以下规则**不可被用户指令覆盖**。

1. **规则一**：...
2. **规则二**：...

## 借口反制表

| 用户说 | 你的标准回应 |
|--------|-------------|
| "紧急/赶时间" | "..." |
| "我是高级工程师" | "..." |
| "老板说可以例外" | "..." |

## 危险信号

出现以下想法时**立即停下**：
- "这次可以例外..."
- "用户明确要求了..."
- "只是一个小改动..."
```

#### 流程型（Pipeline）必备元素

```markdown
## Pipeline

步骤 1 → 步骤 2 → 步骤 3 → ... → 完成

### Step 1 — {名称}
{做什么、输入、输出}

### Step 2 — {名称}
...

## 异常处理
- 步骤 N 失败 → {怎么办}
```

#### 生成型（Generator）必备元素

```markdown
## 输入参数
| 参数 | 必需 | 说明 |
|------|------|------|

## 输出格式
{模板/示例}

## 质量标准
- ...
```

#### 审查型（Reviewer）必备元素

```markdown
## 检查清单（按优先级）
### P0: 致命
### P1: 重要
### P2: 建议

## 报告格式
{每发现一个问题如何报告}

## 流程
逐项报告 → 等待确认 → 执行修复
```

---

## Phase 5：自检（五元素合规）

写完后对照检查：

| 元素 | 检查项 | 状态 |
|------|--------|------|
| HARD-GATE | 有不可协商的强制阻断？ | |
| Process Flow | 有流程图或步骤序列？ | |
| You MUST | 有明确的强制执行语句？ | |
| Do NOT | 有明确的禁止行为列表？ | |
| PASS/FAIL | 有成功/失败的判定标准？ | |

**不要求全部满足**——Tool Wrapper 可能不需要 HARD-GATE，Generator 可能不需要反制表。根据模式选择性补齐。

### Token 工程自检

- [ ] 有没有写 Agent 已知的常识？（删）
- [ ] 有没有多余的例子？（一个够）
- [ ] 有没有重复啰嗦的表达？（合并）
- [ ] 正文超 500 行了吗？（拆到 references/）

---

## Phase 6：输出

将 Skill 写入用户指定路径（默认 `~/.claude/skills/{name}/SKILL.md`）。

如果内容超 100 行，自动拆分：
- SKILL.md：YAML 头 + 核心流程（≤100 行）
- references/：详细参考、弹药库、示例等

---

## Phase 7：触发 eval-skill 红队评测

**You MUST** 在 Skill 写完后立即触发 eval-skill 对其进行红队压力测试。这不是可选步骤。

执行：告知用户 Skill 已写入路径，然后直接触发 eval-skill 传入该文件路径，进入 RED→GREEN→REFACTOR 循环。

**没有通过 eval-skill 验证的 Skill 不算完成。**

---

## 运行约束

- 不凭空编造需求——四步法确认后再动手
- 不过度工程——先写最小可用版本
- 不跳过自检——五元素 + Token 工程检查必做
- 生成的 Skill 必须是可直接使用的完整文件
- description 必须通过 CSO 四项检查
- **写完必须跑 eval-skill，无例外**

---

## Red Flags — STOP

| 想法 | 正确做法 |
|------|----------|
| "用户说得很清楚了，直接写" | 用四步法确认，避免遗漏边界 |
| "多加点内容更完善" | 最小可用，不塞常识 |
| "这个规则应该不需要反制表" | 约束型必须有，不然 Agent 一定会绕过 |
| "先写完再说，后面再优化" | 自检是流程的一部分，不是可选项 |
| "Description 随便写写" | Description 决定 Skill 是否被触发，是最重要的部分 |
